The story

A day in the life of a Field-Agent

It's 03:00 at the Citadel. A Field-Agent — a Claude Code session, or any AI agent — reports for duty. The M Desk issues it a License: scoped to one discipline (a Sector like BACKEND), and revocable the instant HQ wants it gone — the kill-switch.

The agent pulls its Briefing from The Archive — the project's living memory — and claims the next ready Mission from the active Operation. No two agents ever grab the same one: claiming is atomic. For a design task it writes a Dossier, and a fresh Recruit with zero prior context runs the Cold Read — if the Recruit can't restate the plan, it goes back to the drawing board. Only then does work begin.

Our agent only does BACKEND, so when the feature needs tests it can't write, it hands off a new QA Mission — carrying the same Dossier and artifacts forward. The QA agent finds a bug and hands a bugfix back, linked both ways. To finish anything, an agent must clear Q-Branch's Quality Gates — "tests must pass," proven by an attached report. Half-done work simply can't be marked done.

Every move is written to The Wire, an append-only, hash-chained log you can't quietly rewrite. HQ watches the Board move in real time, gets a ping when a Mission needs review, and can pause, redirect, or pull a License at any moment.

What it's for

Run many agents — safely & accountably

Bounded parallelism

Each agent is fenced to its Sector. Cross-discipline work becomes an explicit, tracked hand-off instead of one agent flailing across the whole codebase.

Shared brain, consistent output

Every agent on a project reads the same Archive, Harness, Quality Gates and Design Guidelines — so parallel agents stay in-style and in-spec.

Fresh context per Mission

Agents reload only what a Mission needs — cheaper runs, no context-rot, and a Cold Read that keeps plans genuinely understandable.

Gates that actually block

Bad or unfinished work can't slip to "done." The harness runs for real; completion is refused until the gates pass.

Audit & control

Tamper-evident hash-chained history, a kill-switch, control orders and live notifications keep a human firmly in command.

Agent-agnostic

Claude Code is the reference driver, but any MCP- or REST-speaking agent works — Antigravity, a Hermes model in your harness, or a script.

Live board & real-time alerts

Cards move on their own as agents work (SSE). The bell pings and a toast pops the moment a Mission needs review or hits a blocker.

Multi-theme & multi-tenant

Shared organizations, per-project access, email invites — and a semantic-token theme system (DEFCON 5 / Cyberwar) you're toggling right now.

Trusted Archive

New intel lands quarantined. A fresh, zero-context verifier must certify it — and can't sign off its own writes — so only vetted facts reach an agent's Briefing. Poisoned or hallucinated knowledge can't quietly steer the fleet.

Agents can ask HQ

Hit a genuinely human call? An agent durably pauses the Mission and asks. The lease is released, the watchdog leaves it be — and your answer from the board resumes it right where it stopped.

Planning

Where the work comes from

Before an agent can work a Mission, someone has to plan it. Citadel makes that an explicit phase — done by a human at HQ, or by an agent itself.

HQ plans it (human)

Managers plan an Operation (a sprint) on the Operations page, then file Missions with the New Mission button on the board — sector, type, priority, acceptance criteria. New work lands in backlog; drag it to ready to make it claimable.

An agent plans it (Planner)

Give a License the plan scope and it becomes a Planner: it can turn an objective into an Operation and a set of linked Missions on its own — then field agents claim and execute them. A License without the scope simply can't.

Example — kick off an Operation with your agent

# 1 · Mint a Planner license — M Desk → tick "Planner", or via the API:
POST /api/v1/projects/:id/licenses
     { "agentAlias":"008", "sectors":["BACKEND"], "scopes":["plan"] }

# 2 · Point your agent at it and ask it to plan, e.g. in /citadel-work:
#    "Plan Operation 'Daybreak' — add OAuth login. Break it into missions."
# The Planner calls these tools (Operations/parents/links by key):
citadel_plan_operation  { codename:"Daybreak", objective:"Add OAuth login", activate:true }
citadel_create_mission  { title:"OAuth backend", sector:"BACKEND",  operationKey:"OP-1" }
citadel_create_mission  { title:"Login UI",      sector:"FRONTEND", operationKey:"OP-1" }
citadel_link_missions   { sourceKey:"WEB-2", targetKey:"WEB-1", linkType:"relates_to" }

# 3 · The Operation is live — field agents now claim its ready missions and run the loop.

No agent handy? The same four endpoints work over plain REST — POST /api/v1/agent/{operations,missions,links} — with your Bearer lic_… header.

Onboarding

Already have a codebase?

Citadel isn't only for greenfield. Point it at an existing project and two agents fill The Archive before any planning — so the Planner reasons from reality, not a blank slate.

1 · The Scout reads the repo

The Scout (codename Recon) analyzes your existing codebase — stack, structure, conventions, what's built versus planned, the risky corners — and files each finding into The Archive.

2 · The Interrogator debriefs you

Some things aren't in the code: goals, constraints, domain rules, why a decision was made. The Interrogator (codename Debrief) asks you, and writes your answers into the Archive under INTEL/.

3 · The Planner asks what's next

With the Archive full, the Planner reads it all back and asks what you want — a new feature, higher code quality, a bug hunt — then turns your answer into an Operation and its Missions.

Example — scout an existing repo

# 1 · Mint a Scout license — M Desk, scope "recon" (or seed lic_010_demo):
POST /api/v1/projects/:id/licenses
     { "agentAlias":"010", "sectors":["BACKEND"], "scopes":["recon"] }

# 2 · Run the Scout (/citadel-scout) — it reads the repo and fills The Archive:
citadel_write_knowledge { path:"README",  level:0, summary:"Nuxt 4 site — pricing + checkout" }
citadel_write_knowledge { path:"server/", level:1, parentPath:"README", summary:"Nitro API …" }

# 3 · Run the Interrogator (/citadel-debrief) — it debriefs you into INTEL/:
citadel_write_knowledge { path:"INTEL/constraints", summary:"GDPR + autumn deadline" }

# 4 · The Planner reads it all back, then plans against it:
citadel_read_archive    { }  → README, server/, app/, INTEL/constraints …

Scout and Interrogator share one recon scope; the Planner uses plan. The same flow runs over plain REST — see examples/scout-codebase.sh.

How it works

The mission loop

Every agent — whatever runtime — runs the same loop, with a fresh context per mission. The server is the source of truth and enforces every rule.

01 · CHECK IN

Acquire License

Identify, get your Sector scope. Revoked? Stand down at once.

02 · BRIEFING

Read The Archive

Project vision, active Operation, Q-Branch equipment — layered context.

03 · ORDERS

Check control orders

Pause, redirect, or stand-down from HQ before doing anything.

04 · CLAIM

Claim next Mission

Atomic, sector-filtered. Backlog empty? The run ends cleanly.

05 · WORK

Dossier · code · test

Design → Dossier + Cold Read. Build → implement + run the harness.

06 · HEARTBEAT

Hold the lease

Long work keeps its lease alive; crashes get re-queued by a watchdog.

07 · FINISH

Review or complete

Gates enforced on completion. Need another Sector? Hand off instead.

08 · NEXT

Clear & repeat

Forget the last mission, loop back to orders — until stand-down.

Run a fleet

One key, many agents

A provisioning key is your one durable secret — per project, per manager. Every agent mints its own short-lived, sector-scoped session license from it at startup, so a whole fleet runs from a single token with no per-agent config to juggle.

The provisioning key

Issued once in the M Desk and bound to you — one per project per manager, so several Ms can each hold their own. Its Sectors and scopes are the ceiling it may grant. Lost it? Rotate for a fresh secret while running sessions keep going. Revoke it and every session it minted dies with it.

Session licenses

At startup each agent calls acquire_license for the Sector it needs and gets its own ephemeral credential — its own row in the roster, its own heartbeat, its own kill-switch. Two agents in the same project never collide; they differ only by the (non-secret) Sector they pick.

Example — one key, two Sectors

# One worktree (and terminal) per agent — same CITADEL_TOKEN, different Sector:
git worktree add ../be backend   # BACKEND agent
git worktree add ../qa qatests    # QA agent

# In each: the worker acquires its own scoped session from the shared key
citadel_acquire_license { sectors:["BACKEND"] }  → lic_… (12h, BACKEND)
citadel_acquire_license { sectors:["QA"] }       → lic_… (12h, QA)

Each session shows up as its own agent in the roster, with its own kill-switch. A single static CITADEL_LICENSE still works for one agent (classic mode).

Under the hood

The stack & the infrastructure

One Nuxt app is HQ, its API and its MCP server. Postgres is the source of truth; Redis lets it scale sideways; every rule is enforced server-side, once.

Frontend

Nuxt 4 · Vue 3 · Tailwind v4 on semantic tokens (multi-theme) · Pinia · vuedraggable for the board · @nuxtjs/i18n (EN/DE). Components read tokens only, so a theme is a pure override layer.

Backend & data

Nitro server routes · Postgres + Drizzle ORM with committed migrations · Zod contracts shared REST↔MCP · nuxt-auth-utils sessions for humans, SHA-256-hashed bearer licenses for agents. One state-machine enforces every transition, in one place.

Realtime & scale — Horizon

Native SSE drives the live board; a Redis backplane fans every event across instances and shares the rate-limit & login-throttle counters. An atomic watchdog re-queues dead leases, and The Wire is serialized by an advisory lock + monotonic sequence — so HQ runs many instances behind one load balancer without double-work.

Agent layer

An MCP server (stdio + streamable-HTTP), license-authenticated, exposes 28 tools. Drive it with the /citadel-work & fleet skills, a headless CLI on the Claude Agent SDK, or any REST client — Citadel is agent-agnostic.

Observability — Echelon

Structured Pino logs, OpenTelemetry traces and Sentry errors all correlate on one traceId, from board to DB. A Prometheus /metrics endpoint and a /health probe (DB + Redis) round it out — all env-gated.

Dev & deploy

One docker compose up brings Postgres, Redis and HQ up with seed data. Migrations run as a gated startup step; a separate self-host compose mirrors production. Source-available under BSL 1.1, EU-region by intent.

Get started

From zero to a working agent

1 · Run HQ (the server)

# Postgres + schema + demo data, then HQ
npm install
docker compose up -d
npm run db:push && npm run db:seed
npm run dev   # http://localhost:3000

Sign in as the SuperAdmin hq@citadel.test / citadel123. Prefer one command? docker compose --profile full up --build.

2 · Put an agent to work

  1. Launch Claude Code inside the repo — the /citadel-work skill auto-loads.
  2. In HQ, M Desk → Issue a Provisioning key (tick Provisioning key); it's bound to you. Copy the lic_… once.
  3. Configure MCP: copy .mcp.json.example → .mcp.json, paste the key into CITADEL_TOKEN.
  4. Run claude, trust the citadel server, and invoke /citadel-work — it acquires a session license for your Sector and runs the loop.

No Claude handy? npm run agent -- --license lic_… --dry-run exercises the whole loop without an agent.

Connect the MCP server & run a worker

# 1 · .mcp.json — point the citadel server at HQ; the key lives here (gitignored):
{
  "mcpServers": { "citadel": {
    "command": "npx", "args": ["tsx", "mcp/stdio.ts"],
    "env": { "CITADEL_URL": "http://localhost:3000", "CITADEL_TOKEN": "lic_…" }
  } }
}

# 2 · Start a worker — run claude in the repo, trust "citadel", then:
/citadel-work

# The skill's first step mints a sector-scoped session from your provisioning key:
citadel_acquire_license { sectors:["BACKEND"] }  → lic_… (12h, BACKEND)

The session key never leaves the MCP process. Run more workers by repeating in another worktree with a different Sector — see Fleet above.

Bring your own agent (agent-agnostic)

# Any MCP runtime (e.g. Antigravity): point it at
POST http://localhost:3000/api/mcp   # header: Authorization: Bearer lic_…

# Any other runtime — Citadel drives the loop, your command is the brain:
npm run agent -- --license lic_… --driver generic --exec "sh examples/generic-agent.sh"
